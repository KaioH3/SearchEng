SearchEng - Meta-Search Engine em Go
=====================================
Resumo da implementação - 2026-03-03
Atualizado: 2026-03-05 (v2: anti-bloqueio)

## O que foi feito

### 1. Setup do projeto
- Go 1.24.1 instalado em ~/.local/go (adicionado ao PATH via .zshrc)
- Módulo Go inicializado: github.com/usuario/searcheng
- Dependência: golang.org/x/net v0.51.0 (parser HTML)
- Estrutura de diretórios: engine/, api/, config/

### 2. Tipos base (engine/result.go, engine/provider.go)
- Result: Title, URL, Snippet, Source, Score (interno para ranking)
- SearchResponse: Query, Results, Duration, Page
- Interface Provider: Search(query, page) e Name()

### 3. Providers implementados (engine/)
- DuckDuckGo (duckduckgo.go): scraping HTML de html.duckduckgo.com/html/
  - Extrai resultados de links com classe "result__a" e snippets "result__snippet"
  - Resolve URLs de redirect do DDG (parâmetro uddg)
- Google (google.go): scraping HTML de google.com/search
  - Busca resultados em divs com classe "g", títulos em <h3>, snippets em "VwiC3b"
- Bing (bing.go): scraping HTML de bing.com/search
  - Resultados em <li class="b_algo">, títulos em <h2><a>, snippets em <div class="b_caption"><p>
- Brave (brave.go): API REST oficial (free tier 2000 queries/mês)
  - Header X-Subscription-Token para autenticação
  - Requer BRAVE_API_KEY no environment

### 4. Engine/Aggregator (engine/engine.go)
- Busca paralela: goroutines para cada provider com sync.WaitGroup
- Timeout global configurável (default 5s)
- Deduplicação por URL normalizada (lowercase host, sem trailing slash, sem fragment)
- Merge ranking: score baseado em posição + bonus de +10 para resultados em múltiplos buscadores
- Mantém o snippet mais longo quando há duplicatas

### 5. Config (config/config.go)
- Carrega de variáveis de ambiente:
  - BRAVE_API_KEY, SEARCHENG_PORT (8080), SEARCHENG_TIMEOUT (5s), SEARCHENG_MAX_RESULTS (20)
  - SEARCHENG_MAX_RETRIES (2), SEARCHENG_RETRY_DELAY (500ms)

### 6. CLI (main.go)
- Subcomando "search": busca e exibe resultados formatados no terminal
  - Flags: --providers=, --max-results=
- Subcomando "serve": inicia servidor REST
  - Flags: --port=, --providers=

### 7. API REST (api/server.go)
- GET /search?q=query&page=1 → JSON com resultados
- GET /health → {"status":"ok"}
- Sem autenticação (uso local)

### 8. Anti-bloqueio (engine/httpclient.go)
- Pool de 6 User-Agents modernos compartilhado por todos os providers de scraping
- Headers realistas (Accept, Accept-Language, DNT, Connection, Upgrade-Insecure-Requests)
- Retry com backoff exponencial + jitter em 429, 503, 5xx e erros de rede
- Rate limiting por provider: DDG 10/min, Google 5/min, Bing 10/min
- Cookie jar para scraping providers (mantém sessão entre requests)
- Stack de transports: retry -> rate limit -> default transport

## Análise de qualidade do código (2026-03-05)

### Pontos fortes
- Arquitetura limpa: interface Provider bem definida, fácil adicionar novos buscadores
- Busca paralela com goroutines e WaitGroup → bom desempenho
- Deduplicação inteligente com normalização de URL
- Timeout global protege contra providers lentos
- Código Go idiomático, sem dependências desnecessárias

### Melhorias sugeridas para qualidade de pesquisa

1. **Anti-bloqueio / Resiliência** (IMPLEMENTADO em 2026-03-05)
   - [x] Rotação de User-Agent: pool compartilhado de 6 UAs modernos para todos os providers
   - [ ] Pool de proxies: rodar requests por SOCKS5/HTTP proxies para evitar rate-limiting
   - [x] Retry com backoff exponencial: status 429/503/5xx + erros de rede, configurável via env
   - [x] Headers realistas: Accept, Accept-Language, DNT, Connection, Upgrade-Insecure-Requests
   - [x] Cookie jar: manter cookies entre requests para sessões mais naturais
   - [x] Rate limiting interno: DDG 10/min, Google 5/min, Bing 10/min

2. **Qualidade do ranking**
   - Ranking atual é simples (posição + bonus multi-source)
   - Melhorias possíveis:
     - TF-IDF ou BM25 entre query e snippet para relevância textual
     - Boost por domínio confiável (wikipedia, docs oficiais, etc.)
     - Penalizar resultados sem snippet
     - Considerar freshness (data do resultado quando disponível)
     - Normalização de scores entre providers (cada um retorna em ordem diferente)

3. **Algoritmo genético para auto-otimização (ideia avançada)**
   - Cromossomo = pesos do ranking (peso_posição, peso_multi_source, peso_snippet_length, etc.)
   - Fitness = click-through rate ou satisfação do usuário (precisa de feedback loop)
   - Operações: crossover dos pesos, mutação aleatória, seleção por torneio
   - Funcionaria assim:
     a) Gerar população inicial de configurações de ranking
     b) Para cada query, aplicar N configurações e medir qual ranking é "melhor"
     c) Evoluir pesos ao longo do tempo
   - Desafio: precisa de métrica de qualidade (ex: usuário clica no 1º resultado = bom)
   - Alternativa mais simples: A/B testing dos pesos com logging de cliques

4. **Mais providers**
   - SearXNG (meta-search open source, muitas instâncias públicas)
   - Mojeek (search engine independente com API)
   - Yandex (bom para resultados internacionais)
   - Qwant (europeu, boa privacidade)

5. **Cache local**
   - Cache de resultados por query+provider com TTL (ex: 1h)
   - Reduz requests e melhora latência para queries repetidas
   - Pode ser in-memory (map com mutex) ou SQLite/BoltDB

6. **Scraping mais robusto**
   - Google muda HTML frequentemente → ter fallback selectors
   - Detectar CAPTCHAs e avisar o usuário
   - Headless browser (chromedp/rod) como fallback para JavaScript-heavy pages
